Introduction

What is a Network Intrusion Detection System?

A Network Intrusion Detection System (NIDS) monitors network traffic in real time, analyzing packet data to identify suspicious or malicious behavior. It sits passively on the network, watching every connection and flagging anomalies before they escalate into breaches.

Signature-Based Detection
Matches traffic patterns against a database of known attack signatures. Fast and accurate for known threats but entirely blind to new, unseen attack variants (zero-day attacks).
🧠
Anomaly-Based (AI) Detection
Learns what "normal" traffic looks like and flags statistical deviations. Can detect novel and evolving attacks even when they don't match any known signature.
📋
Policy/Rule-Based Detection
Enforces manually-written IF-ELSE rules (e.g., "if more than 400 connections in 2 seconds, flag as DoS"). Simple to understand but requires experts to write and maintain rules continuously.
AI Model

How the AI Model Works — Random Forest

This NIDS uses a Random Forest Classifier trained on the NSL-KDD dataset. Random Forest is an ensemble of decision trees, each trained on a random subset of data, whose majority vote determines the final classification.

1

Raw Network Packet Arrives

Every TCP/UDP/ICMP connection is captured and its 18 statistical features are extracted — including byte rates, connection counts, error rates, login failures, and service ratios.

2

Feature Scaling (StandardScaler)

All feature values are normalized using Z-score standardization (mean=0, std=1). This ensures features with large ranges (like src_bytes) don't dominate features with smaller scales (like error rates).

3

Voted Classification by 100 Decision Trees

Each of the 100 decision trees in the Random Forest independently classifies the packet. Each tree was trained on a different random subset of the training data and considers a random subset of features at each split — preventing overfitting.

4

Majority Vote Determines Final Label

The class that receives the most votes among all 100 trees wins. The confidence score is the fraction of trees that agreed (e.g., "92% confidence = 92/100 trees voted for DoS"). The packet is labelled: Normal, DoS, PortScan, or BruteForce.

5

Alert Generated (if threat detected)

If the predicted label is not Normal, an alert entry is appended to the Security Alert Log on the dashboard with timestamp, severity, and description — without any popup interruption.

# Simplified: how a packet is classified at runtime
vec = [src_bytes, dst_bytes, count, serror_rate, ...] # 18 features
vec_scaled = scaler.transform(vec) # normalize
votes = [tree.predict(vec_scaled) for tree in forest] # 100 trees vote
label = majority_vote(votes) # e.g. "DoS"
confidence = votes.count(label) / 100 # e.g. 0.87
Why Random Forest?
It handles mixed feature types, is resistant to overfitting via ensemble voting, provides calibrated confidence scores, and performs well even with imbalanced class distributions — all critical for NIDS.
NSL-KDD Dataset
6,400 labeled network connections used for training. Includes Normal, DoS, PortScan, and BruteForce samples. Each record has 18 numeric features extracted from raw packet-level data.
Key Features Used
src_bytes (data volume), count (connection rate), serror_rate (SYN errors), diff_srv_rate (service diversity), num_failed_logins (authentication failures), and 13 others.
Rule-Based System

How Rule-Based Detection Works

Rule-based NIDS use manually-coded IF-ELSE conditions written by security experts. While they are transparent and fast, they can only catch attacks that exactly match their pre-written rules.

# The 3-rule system used in this NIDS (deliberately limited)

if count > 400 AND serror_rate > 0.85:
label = "DoS" # Only catches extreme, high-volume floods

elif diff_srv_rate > 0.85 AND rerror_rate > 0.80:
label = "PortScan" # Misses stealthy/slow scans

elif num_failed_logins > 5 AND logged_in == 0:
label = "BruteForce" # Misses distributed or slow brute force

else:
label = "Normal" # Everything else passes — including moderate attacks!
Advantages
Zero false positives for known, extreme attack patterns
Fully explainable — any analyst can read the rules
No training data required
Very fast at runtime — just a few IF checks
Limitations
Can only detect attacks whose features exceed hard-coded thresholds
Completely misses "moderate" attacks that stay below thresholds
Requires constant manual updates as attackers adapt
Does not generalize to new attack types

Attack Types

Network Attacks Detected by This System

This NIDS detects three major categories of network attacks alongside normal traffic. Each attack type has distinct network-level signatures and real-world consequences.

Denial of Service (DoS)

Availability attack — Severity: HIGH
What it is
An attacker floods a server with overwhelming traffic — thousands of SYN packets per second — exhausting CPU, memory, or bandwidth until the server becomes unavailable to legitimate users.
Harm caused
Service outages (websites go offline), financial losses from downtime, and reputational damage. DDoS attacks on banks can cost millions per hour.
Network signs
Very high count (150–500+), high SYN error rate (serror_rate > 0.5), minimal dst_bytes (server can't respond).
Rule vs AI
Rule only flags count > 400. AI detects moderate floods at count 150–400 that rules miss entirely.
Prevention
Rate limiting and SYN cookies at the firewall level
CDN-based traffic scrubbing (Cloudflare, Akamai)
Anycast network diffusion to distribute load
Real-time NIDS alert triggering automated IP blocking

Port Scanning

Reconnaissance attack — Severity: MEDIUM
What it is
An attacker systematically probes ports on a target host to discover which services are running (SSH on 22, HTTP on 80, MySQL on 3306, etc.) and identify potential entry points for a deeper attack.
Harm caused
By itself it does no direct damage, but it is always the first stage of a targeted attack. A port scan map is used to plan the actual exploit.
Network signs
High diff_srv_rate (many different services probed), high rerror_rate (rejected connections), low src_bytes, minimal duration.
Rule vs AI
Stealthy scans with diff_srv_rate 0.60–0.85 and rerror_rate 0.55–0.80 evade the rule but are caught by the AI model.
Prevention
Firewall rules blocking unused ports
Network segmentation to limit scan surface
Port-knocking or honeypot ports to detect scanners early
NIDS alert on rapid cross-port connection attempts

Brute Force Attack

Authentication attack — Severity: HIGH
What it is
An attacker tries thousands of username and password combinations automatically until they find one that works. Modern attacks are often distributed across many IPs to avoid rate-limit detection.
Harm caused
Account compromise, data theft, ransomware installation, lateral movement through the network, and full system takeover in worst cases.
Network signs
Elevated num_failed_logins (1–6+), high rerror_rate, repeated connections to the same service (same_srv_rate near 1.0).
Rule vs AI
Rule only triggers at failed_logins > 5. Distributed attacks (1–2 failures per IP) evade the rule but AI detects the combined pattern.
Prevention
Multi-factor authentication (MFA)
Account lockout policies (lock after 5 failed attempts)
CAPTCHA on login forms
IP reputation blocking and geo-based access rules
Defense Strategies

How to Defend Your Network

A modern defense-in-depth strategy combines multiple layers. No single tool is sufficient — the goal is to detect and contain attacks at each layer before they reach critical assets.

1

Perimeter Defense — Firewall and Rate Limiting

The first layer. Stateful firewalls block unauthorized traffic by port and protocol. Rate limiters cap connection rates per IP, preventing volumetric DoS floods from even reaching your servers.

2

Intrusion Detection and Prevention (NIDS/NIPS)

AI-powered NIDS like this one analyze traffic patterns in real time and generate alerts. A NIPS (prevention system) goes further by automatically blocking suspicious IPs when confidence exceeds a threshold.

3

Authentication Hardening

Enforce MFA on all user accounts, use SSH keys instead of passwords for server access, set strong account lockout policies (5 failures = 15-minute lockout), and use password managers to eliminate weak credentials.

4

Network Segmentation and Zero Trust

Divide your network into isolated segments. A compromised machine in one segment cannot reach critical assets in another. Zero Trust means every user and device must authenticate even inside the network perimeter.

5

Continuous Monitoring and Incident Response

Alerts are only useful if someone acts on them. Establish a Security Operations Center (SOC) with defined incident response playbooks. Regularly review NIDS logs and tune detection thresholds to reduce false positives over time.
